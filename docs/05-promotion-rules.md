# Promotion Rules

Atlas Memory should not turn every observation into a permanent rule. Promote notes gradually.

## 1. Raw note or session entry

Use for fresh observations, command results, open questions, and partial understanding.

Common locations:

- `session/Next session handoff.md`
- `queue/00-work-queue.md`
- A dated note under `evidence/`

Label uncertain claims as `draft`, `unverified`, or `stale` when appropriate.

## 2. Lesson learned

Use when a raw note has proved useful and should be easier to find.

Common locations:

- A relevant doc under `docs/`
- An evidence note with a short conclusion
- The Atlas start-here map as a pointer

Keep the lesson specific. Include evidence when the lesson depends on investigation.

## 3. Promoted active rule

Use when a lesson should guide future behavior in the repo.

Common locations:

- A canonical doc under `docs/`
- A safe-boundary note
- A workflow section in the Atlas map

Active rules should be clear, current, and easy to validate.

## 4. `AGENTS.md` rule

Use only when the rule is stable, high-value, and likely to apply across many future Codex sessions.

Good candidates:

- Required read-first files.
- Public wording constraints.
- Safety boundaries.
- Required branch, validation, commit, or push workflow.

Poor candidates:

- One-time task notes.
- Personal preferences that do not affect the repo.
- Unverified assumptions.
- Details better kept in canonical docs.

## Demotion

If a rule stops being true, demote or remove it. Mark historical notes as `historical` instead of leaving old guidance active.
